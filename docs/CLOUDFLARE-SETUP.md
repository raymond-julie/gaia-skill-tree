# Cloudflare Setup — Repo-bound badge validation

This document explains the **Worker** that gates Gaia README badges on a
`?repo=` query parameter, and how to verify it locally and in production.

## What this does

- Every badge under `https://gaia.tiongson.co/badges/<handle>/<file>.svg`
  validates the `?repo=<owner>/<repo>` query (the badges page generates these
  URLs). The site Worker (`worker/index.js`) looks `<owner>/<repo>` up in
  `docs/badges/registry.json` and serves the **validating badge**
  (`docs/badges/not-found.svg` — a faint *"validating…"* SVG) when the repo
  isn't registered to that handle. A matching repo (or no `?repo=` at all, for
  back-compat) serves the real badge.
- 24-hour `max-age` so GitHub camo (which keys on the full URL incl. query)
  caches each `(handle, file, repo)` image for ~24h.

## The design — relocated assets, no `run_worker_first`

This project deploys as a **Cloudflare Worker with Static Assets**
(`wrangler deploy`, `[assets] directory = "docs"`). Two hard facts shaped the
design:

1. **`functions/` is Pages-only** — a Pages Function is never deployed by
   `wrangler deploy`.
2. **Static assets are served before the Worker.** `run_worker_first` is
   supposed to flip that, but it proved unreliable across this project's deploy
   pipeline (silently not applied, so the worker only ran as the no-asset
   fallback and real badges bypassed validation).

So rather than depend on `run_worker_first`, we **remove the conflict**:

- `generateBadges.py` writes the real SVGs to **`docs/badges/_assets/<handle>/`**
  (a 3-segment path).
- The public badge URL stays **`/badges/<handle>/<file>.svg`** (2-segment),
  which now has **no static asset**.
- The Worker therefore runs as the normal **no-asset fallback** for every
  `/badges/<handle>/<file>.svg` request — the one behaviour that works reliably
  in production — validates `?repo=`, and serves the real SVG from the
  `_assets` source (or `not-found.svg` on a mismatch).

`wrangler.toml` needs only `main` + the asset binding (no `run_worker_first`):

```toml
name = "gaia-skill-tree"
main = "worker/index.js"
compatibility_date = "2026-05-22"

[assets]
directory = "docs"
binding = "ASSETS"
```

### Routing summary

| Path | Served by |
|---|---|
| `/badges/<handle>/<file>.svg` | **Worker** (no static asset here → fallback) — validated |
| `/badges/_assets/<handle>/<file>.svg` | static (the real SVGs; worker reads these) |
| `/badges/{powered-by-gaia,not-found}.svg` | static (1-segment) |
| `/badges/samples/*` | static |
| `/badges/registry.json`, `/badges/`, everything else | static |

### Caching (handled in the worker, no CDN cache rule needed)

Badge responses are generated by the Worker (not served as static assets), so
Cloudflare does not edge-cache them — the Worker runs per request and each
`?repo=` is evaluated fresh, so different repos never collide on one cached
image. The 24h `max-age` is for GitHub camo's per-URL cache.

## What's in the repo

| Path | Role |
|---|---|
| `wrangler.toml` | Worker entry (`main`) + `[assets]` binding. Deploy root is `/`. |
| `worker/index.js` | The validator. Handles `/badges/<handle>/<file>.svg`, serves from `_assets`. |
| `docs/badges/_assets/<handle>/` | The real per-contributor SVGs (generated). |
| `docs/badges/registry.json` | Approved-repos lookup the Worker reads. Schema `gaia-badges-registry/2`. |
| `docs/badges/not-found.svg` | The "validating…" badge served on `?repo=` mismatch. |
| `docs/wrangler.toml` | Secondary/fallback config (assets only). Not used by the configured deploy. |
| `tests/test_badges_function.md` | Manual smoke-test list runnable via `wrangler dev`. |

## Local verification

```bash
npm install -g wrangler
wrangler dev            # reads ./wrangler.toml; serves on :8787
```

```bash
# 1 — back-compat (no ?repo=) → real badge, x-gaia-badge-state: valid
curl -sI http://localhost:8787/badges/mbtiongson1/handle.svg | grep -i 'x-gaia\|cache-control'

# 3 — wrong repo → validating badge
curl -s 'http://localhost:8787/badges/mbtiongson1/handle.svg?repo=evil/repo' | grep -o 'validating'
curl -sI 'http://localhost:8787/badges/mbtiongson1/handle.svg?repo=evil/repo' | grep -i 'x-gaia-badge-state'
```

`x-gaia-badge-state`: `valid` (real badge) | `validating` (mismatch, faint badge).

## Production verification (after deploy)

The canonical endpoint is the Worker's URL
(`https://gaia-skill-tree.<account>.workers.dev`); the custom domain is
optional.

```bash
H=https://gaia-skill-tree.marco-tngsn.workers.dev
curl -s "$H/badges/mbtiongson1/handle.svg?repo=mbtiongson1/gaia-skill-tree" | grep -o '<svg' >/dev/null && echo "valid path OK"
curl -s "$H/badges/mbtiongson1/handle.svg?repo=evil/repo" | grep -o 'validating'   # must print 'validating'
```

If case #3 returns the real SVG, the Worker isn't handling the route — confirm
`docs/badges/<handle>/` has **no** static SVGs (they must live under
`docs/badges/_assets/<handle>/`) and that the deploy used the repo-root
`wrangler.toml`.

## How invalidation works

- **In-process registry cache:** the Worker holds the parsed `registry.json` in
  module scope for 24h. A redeploy resets it (fresh bundle), so a newly-added
  repo is live immediately for direct hits.
- **GitHub camo:** ~24h regardless. Users can force-refresh their README to bump it.

## Rollback

```bash
# Remove the worker entry; keep assets serving (validation off).
# wrangler.toml → delete `main`, and revert generateBadges.py to write
# per-contributor SVGs back to docs/badges/<handle>/, then `gaia docs build`.
git rm worker/index.js
git commit -m "revert: badge validation worker"
git push
```

## Known limitations / future work

- **No Referer-based validation.** GitHub camo strips `Referer`, so we only
  check the `?repo=` string against the approved list — a determined actor can
  copy a badge URL, or hit the internal `/badges/_assets/...` path directly.
  Tracked under #155 (the real fix is GitHub OAuth → signed token-bound rendering).
- **Analytics — Path A is live.** The Worker emits one structured log line per
  request via `console.log`; tail with `wrangler tail`. Schema + jq queries in
  `tests/test_badges_function.md` § "Structured log lines". Path B (Workers
  Analytics Engine) is tracked under #455.
