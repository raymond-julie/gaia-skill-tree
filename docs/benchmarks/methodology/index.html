Methodology · Frozen post-merge

How a score becomes evidence.

Every benchmark row that touches a Gaia badge carries a public commit SHA, two SHA-256 fingerprints, and a provenance label from a frozen enum. Self-attestation is rejected at the schema. Citations render, but they never count.

01Provenance

Every benchmark-result evidence row carries a provenance label from a frozen enum of four values. Two of them contribute to Trust Magnitude; two do not.

ValueMeaningTM
verifier-attested A 4★+ Verifier co-signed the score. Attestor is the verifier's GitHub handle. Counted
ci-reproduced A GitHub Actions workflow ran the harness on a public commit. Attestor is <workflow-url>@<sha>. Counted
mirrored Citation-only copy from an external leaderboard. Attestor is the upstream URL. Excluded
pending Freshly filed row awaiting promotion. Excluded

self-attested is intentionally absent and forever rejected — both at the schema and in scripts/validate.py. A self-run score enters as pending; a Verifier or CI job promotes it before merge.

pending is informational under lax validation but a hard error under --strict, which the validator auto-enables when the base ref is main. Pending rows can never survive a merge into main.

Two hashes and a public URL. That is the entire contract.

02Reproducibility fingerprint

Two SHA-256 hashes are required on every row:

  • datasetHash — SHA-256 of the raw dataset. A revision produces a new hash and therefore a new row.
  • benchmarkInputHash — SHA-256 of (dataset + prompt template + harness config). Same dataset, different config, different row.

Both are 64-character lowercase hex. Uppercase, short, or non-hex values are rejected at both the CLI and schema layers. The harness is responsible for emitting them; W2a ships the scaffolding, W2b (issue #905) ships the first real run.py.

03Trust Magnitude contribution

The benchmark-result evidence type is one of the highest-weight signals in the registry:

  • weight: 1.4
  • gradeCeiling: S
  • allowedLayers: [generic, named]
  • inheritMultiplier: 0.15
  • Magnitude = percentile (0..100), when provided.
  • Freshness: 50%/year decay (half-life ≈ 1 year).

Mirrored and pending rows are excluded from the sum entirely via computeArtifactScoreOrNone returning None, matching the null-on-derank contract used for deranked verifier-attestation rows. Their grade field is stripped at CLI write time so they are never visually presented as if they were graded.

04Adding a new benchmark

  1. Author the harness. Deterministic Python, pinned deps, emits results JSON with score, unit, runAt, datasetHash, and benchmarkInputHash. Lands at scripts/benchmarks/<id>/run.py.
  2. Author the CI workflow. .github/workflows/benchmark-<id>-ci.yml runs run.py on the submitted commit and, on success, writes an evidence row via gaia dev evidence with --provenance ci-reproduced and --attestor <workflow-url>@<sha>.
  3. Register the benchmark id. Add a landing document at docs/benchmarks/<id>/ (see humaneval-v1).
  4. Wire the leaderboard. Point at docs/api/v1/benchmarks/<id>.json.

The benchmarkId field is semver-ish: <name>@<version> or <name>/<subset>@<version>. Examples: humaneval@v1.0, humaneval/python@v1.0, swe-bench_verified@1.0, mmlu-5shot@2024-03.

A rename is a major schema bump. Names in this document are load-bearing.

05Frozen invariants

  • Field names and shape of benchmark-result are frozen — a rename or removal is a major schema bump (v7+).
  • The provenance enum is frozen.
  • The unit enum (pct, pass@1, pass@10, bleu, f1, accuracy, elo, raw) is frozen.
  • Mirrored rows are forever excluded from Trust Magnitude.
  • Self-attested provenance is forever rejected.

Any change requires a Splurge-tier RFC.
