Evidence Classes

Stars in Gaia are never declared — they are derived from evidence. This page explains the legacy Class system, the new Evidence Type + Grade model, verification states, and how evidence gates each star level.

Overview

Every skill in the Gaia registry has a stars level (0★ to 6★). That level is determined entirely by the evidence attached to it — the stronger and more numerous the demonstrations, the higher the skill can rank up. No star is awarded by fiat.

Gaia is in the middle of a transition from a single-axis Evidence Class (C / B / A) to a two-axis model of Evidence Type (provenance) plus Evidence Grade (quality). Both systems are valid in the current schema; new evidence should use Type + Grade.

⚠ Class letters ≠ Grade letters

The legacy Class A/B and the new Grade A/B look identical but measure different things on incompatible scales. Never equate them. The transition section below explains how they differ.

Legacy Class System

The original evidence model used a single letter that combined provenance and quality into one axis. It is deprecated as of the #646 trust model and will be removed in the next major release; do not add new evidence with only a class field.

C
First sighting
A URL attestation that you or someone else observed this skill being used. No reproduction steps required. Sufficient to name a skill (reach 2★) when attached alongside a valid gaia push batch. The minimum bar for naming.
B
Reproducible
A documented demonstration a peer can follow step-by-step and reproduce. Implies a repo, notebook, or detailed write-up. Stronger than Class C but not yet independently peer-reviewed.
A
Battle-tested / peer-reviewed
Shipped to production, independently validated, or published with peer review. The strongest legacy class. Carries the most weight in the overall trust computation, but new submissions should use Grade S or A instead.
📋 Migration path

The class field stays valid in the schema until the next major release. If you are backfilling old evidence, use --class C|B|A. For all new evidence, omit --class and supply --type + --grade instead.

Evidence Type + Grade

The #646 trust model splits the old single class field into two independent axes. This lets the registry track where a demonstration comes from separately from how strong it is.

Evidence Type — provenance

Evidence Type records where a demonstration originates. Values are kebab-case strings drawn from the evidence.types list in meta.json. New source types can be added to that list without a schema change.

Always write the full phrase Evidence Type — never the bare word "type", which names the Basic / Extra / Unique / Ultimate taxonomy field.

Current provenance values:

arxiv repo github-stars

Evidence Grade — quality

Evidence Grade measures the quality of one demonstration on an S / A / B / C axis. Each grade maps to a metal name: Platinum (S), Gold (A), Silver (B), Bronze (C). A demonstration whose trust number falls below the C threshold is ungraded — on the record but counting toward no star gate.

Grade Scale

Four grades, four metals. Grades are computed at build time from trust numbers — never declared by a contributor.

S
Platinum
trust ≥ 90
Independently reproduced, peer-reviewed, or cited at publication scale.
A
Gold
trust ≥ 80
Shipped to production or validated by a 4★ Verifier.
B
Silver
trust ≥ 60
Reproducible with a working repo or detailed write-up.
C
Bronze
trust ≥ 40
First-sighting attestation — observed but not yet reproduced.

Demonstrations with trust < 40 are ungraded: stored in the registry but excluded from all gate calculations.

Trust Numbers

Each evidence entry has an internal trust number — a 0–100 score the build pipeline computes from factors like citation count, independent reproduction, and Verifier attestation. Trust numbers are never user-facing; the grade they yield (S / A / B / C) is what surfaces in catalogs and the UI.

The thresholds are configured in meta.json under evidence.gradeThresholds and can be tuned by maintainers without a schema change. The current defaults:

meta.json — gradeThresholds (defaults) 
"gradeThresholds": {
  "S": 90,
  "A": 80,
  "B": 60,
  "C": 40
}

Overall Trust Grade

A skill's Overall Trust Grade is the aggregate of its individual Evidence Grades — the accumulation that establishes the capability "beyond reasonable doubt." It is computed from the evidence inventory at build time and is never stored on the registry node itself. It materialises only in generated catalogs (named-skills.json, docs/graph/gaia.json).

Do not confuse a single evidence entry's grade with the skill's Overall Trust Grade — one strong entry does not guarantee a high aggregate.

Verification States

Verification and grading are orthogonal. A piece of evidence can be verified (confirmed as real) independently of how strong it is. Three states exist:

Unverified Verified Disputed
default
Unverified
The default state for every new evidence entry. The evidence is on the record and included in grade calculations, but no 4★ Verifier has confirmed it yet.
✓
Verified — evidence.verified: true
A 4★ Verifier confirmed that the demonstration is real and the URL is accurate. Verification attests authenticity, not strength — a verified Bronze is still Bronze.
✗
Disputed — evidence.disputed: true
A maintainer or Verifier has flagged this entry as questionable. Disputed evidence is excluded from gate calculations until the dispute is resolved.
⚠ Verification ≠ grading

"Verified" means the demonstration is real. Grade measures how strong it is. A verified entry can have any grade; an unverified entry still contributes to trust calculations until disputed. Do not use "verification stars" — verification is not on the stars axis.

CLI Usage

Attaching evidence requires Verifier authorization (4★ named skill in registry/named-skills.json or GAIA_OPERATOR_OVERRIDE=1 in CI).

Add evidence with legacy Class

shell 
# Attach a Class C sighting (legacy — still valid)
gaia dev evidence autonomous-code-review "https://github.com/you/repo" --class C

# Attach a Class B reproducible demo
gaia dev evidence autonomous-code-review "https://arxiv.org/abs/..." --class B

Add evidence with new Type + Grade

shell 
# Preferred: supply type and grade separately
gaia dev evidence autonomous-code-review "https://arxiv.org/abs/..." \
  --type arxiv \
  --grade A

# Repo demo, Silver grade
gaia dev evidence tool-call-chaining "https://github.com/you/agent-demo" \
  --type repo \
  --grade B

Remove evidence

shell 
gaia dev rm-evidence autonomous-code-review "https://github.com/you/repo"

Inspect a skill's evidence

shell 
# Full node detail including evidence entries
gaia dev list --id autonomous-code-review

# Also inspect via gaia skills info
gaia skills info /autonomous-code-review

How Evidence Drives Stars

Stars are gated on evidence: reaching 1★ (Awakened) is free — it marks the first scan detection. Every star above 1★ requires graded evidence on the attached skill node. The naming gate (2★) is the most commonly reached gate for contributors.

Stars Rank Name Evidence gate Notes
0★ Unawakened None Generic registry node, no named contributor.
1★ Awakened None First detected in a gaia scan run.
2★ Named Class C or better (legacy) or any Grade ≥ C (new) Contributor named via gaia push. Requires a valid PR.
3★ Evolved Grade B or better Reproducible demonstration on record.
4★ Hardened Grade A (Gold) or better Contributor becomes a Verifier — can authorize registry mutations.
5★ Transcendent Grade S (Platinum) Peer-reviewed or independently reproduced at scale.
6★ Transcendent ★ Grade S + Verifier attestation The apex — rare, requires multiple independent confirmations.
💡 Starless references

Generic (starless) skill references inherit the highest star among their named variants as their effective rank. Attaching academic-level (Class A or Grade S) evidence directly to a starless node raises its effective rank for all named variants that derive from it.

Checking gate status

shell 
# See your current authorization and Verifier status
gaia whoami

# Run scan to generate fresh promotion candidates
gaia scan

# Promote if a candidate matches the gate
gaia promote autonomous-code-review
