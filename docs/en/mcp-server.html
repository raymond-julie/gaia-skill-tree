MCP Server

Connect Gaia to any MCP-compatible agent — Claude Code, Claude Desktop, Cursor, VS Code, Gemini — and give it native access to skill lookup, fusion suggestions, context scanning, and your personal skill tree.

Overview

@gaia-registry/mcp-server is a Model Context Protocol server that exposes Gaia's registry to your AI agent at inference time. Instead of copying skill IDs into prompts, the agent calls structured tools and receives structured responses — metadata, prerequisites, fusion candidates, and progression stats.

The server is stateless and read-heavy. It fetches gaia.json from GitHub on first request and caches locally with ETag revalidation. Writes (proposing a skill, claiming a fusion) require a GITHUB_TOKEN and open a PR on your behalf.

💡 Quick start

Using Claude Code? Run one command and you're done:

claude mcp add gaia -- npx @gaia-registry/mcp-server

Then ask: "Use gaia_lookup to find the rag-pipeline skill."

Installation

The universal install shape is npx @gaia-registry/mcp-server with an optional GAIA_USER env var set to your GitHub username. Platform specifics follow.

Claude Code (CLI)

One-liner registration — works project-locally or globally:

bash
# project-local (adds to .claude/settings.json)
claude mcp add gaia -- npx @gaia-registry/mcp-server

# global (available in every project)
claude mcp add gaia --global -- npx @gaia-registry/mcp-server

To pass GAIA_USER and GITHUB_TOKEN, edit .claude/settings.json directly:

.claude/settings.json
{
  "mcpServers": {
    "gaia": {
      "command": "npx",
      "args": ["@gaia-registry/mcp-server"],
      "env": {
        "GAIA_USER": "your-github-username",
        "GITHUB_TOKEN": "ghp_..."
      }
    }
  }
}
⚠ Set GAIA_USER in env, not config.json

The MCP server resolves identity from process.cwd(), which may not be your project root when launched by Claude Code. Always set GAIA_USER in the env block above — this is the only reliable override. See Known issues.

Tools

Five tools are registered. All read-only tools work without any env config. gaia_propose requires GITHUB_TOKEN.

gaia_lookup
Look up a skill by ID or fuzzy name. Returns full metadata — type, stars, description, prerequisites, derivatives, evidence entries, and graph context (ancestors, descendants).
query required Skill ID (web-scrape) or fuzzy name ("web scraping").
gaia_suggest
Get fusion recommendations for what you're building. Compares your connected tools and project description against the registry to surface achievable fusions and skills one step away from unlocking.
context optional Array of project signals — descriptions, package names, file patterns.
tools optional MCP tool names currently connected ("web_search", "bash").
gaia_scan_context
Scan the current session's context to detect skills in use. Pass connected tool names and project signals; the server maps them to Gaia skill IDs and reports fusion opportunities. Set deepScan: true to invoke the local gaia CLI for a full source-code scan.
connectedTools optional MCP tool names available in this session.
projectSignals optional Capability descriptions ("cheerio in package.json").
deepScan optional · bool Invoke local gaia scan CLI for full source analysis.
gaia_my_tree
Fetch a user's Gaia skill tree — unlocked skills with star levels, pending fusions, and progression stats. Defaults to the configured GAIA_USER.
username optional GitHub username. Falls back to GAIA_USER env var.
gaia_propose
Claim a skill fusion or propose a novel skill to the registry. Opens a PR on GitHub. Requires GITHUB_TOKEN or GH_TOKEN in the server environment.
skillId optional Existing skill ID to fuse or claim.
name optional Name for a novel skill proposal.
description optional Description of the novel skill (≥ 10 chars).
type optional basic · extra · unique · ultimate
prerequisites optional Prerequisite skill IDs for extra/ultimate proposals.

Resources

Two MCP resources expose raw registry data. Use these when you want the full graph in context rather than a single skill lookup.

URI Contents Notes
gaia://registry Full skill graph — all skills with type, stars, prerequisites, evidence JSON, ETag-cached. Large (hundreds of nodes); prefer gaia_lookup for single skills.
gaia://tree/{username} A user's skill tree — unlocked skills, stars, timeline events Fetched from skill-trees/{username}/skill-tree.json in the registry repo.

Configuration

The server resolves identity in priority order. The first source that returns a non-empty value wins.

PrioritySourceValue
1 — highest GAIA_USER env var Your GitHub username, set in the MCP server's env block.
2 .gaia/config.json (project) Config file in the directory where the server process was spawned.
3 — lowest ~/.gaia/config.json (global) Global fallback; usually set by gaia init --user <username>.

For gaia_propose, set GITHUB_TOKEN or GH_TOKEN to a personal access token with repo scope. Without it, proposal calls will fail with an authentication error.

env block — both vars
"env": {
  "GAIA_USER": "your-github-username",
  "GITHUB_TOKEN": "ghp_..."
}

Prompt examples

Once Gaia is connected, try these in your agent:

  • "What skills do I have? Use gaia_my_tree."
  • "Scan my current context and suggest skill fusions. Use gaia_scan_context."
  • "Look up the rag-pipeline skill in the Gaia registry."
  • "I just built a multi-step web scraper with Playwright. What Gaia skill does that map to?"
  • "What's the upgrade path from web-search to autonomous-research-agent? Use gaia_lookup."
  • "I have web-scrape and parse-html unlocked. What fusions are available? Use gaia_suggest."
  • "Propose a new skill called 'agentic-code-review' based on what I just built. Use gaia_propose."

How it works

On startup the server registers its tools and resources with the MCP SDK, then waits for calls over stdio. Registry data is fetched lazily — the first tool call triggers a gaia.json download from GitHub, which is stored in-memory and revalidated with ETag on subsequent calls.

When gaia_suggest or gaia_scan_context runs, the advisor module maps your connected tool names and project signals to Gaia skill IDs (via a keyword/semantic match), then walks the prerequisite graph to find any Extra or Ultimate skill whose inputs are fully satisfied. Fusion candidates are returned ranked by confidence.

src/
├── index.ts ← server setup, tool + resource registration
├── graph/ ← registry loader, types, search, DAG traversal
├── tools/
│ ├── lookup.ts ← gaia_lookup
│ ├── suggest.ts ← gaia_suggest
│ ├── scanContext.ts ← gaia_scan_context
│ ├── myTree.ts ← gaia_my_tree
│ └── propose.ts ← gaia_propose (writes PR via GitHub API)
├── advisor/ ← skill detection engine, fusion matcher, novelty scorer
├── config/ ← identity resolution, ETag cache
├── resources/ ← gaia://registry + gaia://tree/{username}
└── utils/ ← GitHub API helpers, error handling

Advisor Architecture

The MCP server runs a unified advisor module to scan project context and suggest skill fusions. All advisors inherit from a common base class: AbstractAdvisor<TResult>.

The engine runs three concrete advisor classes:

  • SkillDetector: Inspects connected tool names and project signals. It uses regular expressions to find match patterns and returns a list of detected Basic Skills (○).
  • FusionEngine: Compares owned and detected skills against the prerequisite graph. It identifies Extra Skills (◇) and Ultimate Skills (◆) that you can unlock immediately (ready) or skills that need only one more prerequisite (one_away).
  • NoveltyScorer: Checks proposed skills against the existing registry. It calculates Jaccard similarity based on word tokens. A proposal is marked novel if the similarity score is below 0.3.

Telemetry & Privacy

The Gaia MCP server collects no usage data, telemetry, or analytics. It has zero tracking pixels or phone-home systems.

All skill detection and graph traversal run locally in your environment. The server only communicates with external hosts to fetch the registry data file (gaia.json) or write pull requests when you use gaia_propose.

Known issues

Identity resolution uses process.cwd() — issue #212

src/config/identity.ts resolves .gaia/config.json relative to process.cwd(). When an MCP host (Claude Code, Cursor, VS Code) spawns the server, the working directory may not be your project root.

Impact: gaia_my_tree and gaia_propose return a "no user configured" error even when a project-level config exists.

Workaround: Set GAIA_USER explicitly in the server's env block. The env var is checked before any file-based resolution and is unaffected by CWD.

fix — always set GAIA_USER in env
"env": {
  "GAIA_USER": "your-github-username"   // immune to CWD issues
}
💡 Development workaround

If you're running the server from source (npm run dev), set GAIA_USER in a .env file at the repo root and load it via your preferred env tool. The env var takes priority over all config files.
