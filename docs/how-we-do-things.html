Curation, automation, and review

How We Work

Gaia is maintained as a reviewable skill graph, not a loose wiki. Every change moves through a small path: propose evidence, let automation shape the batch, review the result, then promote only what belongs in the canonical registry.

The Operating Loop

A simple path keeps public contributions readable and keeps the canonical graph intentional.

1. Discover

Find a real capability

Contributors scan their repositories, name skills they actually use, and attach enough context for another person to understand the behavior.

2. Curate

Keep proposals reviewable

Draft records stay in registry-for-review/ so proposed skills can be discussed without rewriting registry/gaia.json.

3. Review

Check evidence and fit

Reviewers look for duplicates, missing prerequisites, unclear names, and weak evidence before anything moves toward the canonical tree.

4. Promote

Merge only settled graph changes

Accepted items are promoted in focused graph PRs so each canonical change has a clear source, rationale, and reviewer trail.

What Automation Does

Automation helps with formatting, triage, and repeat checks. Humans still decide what the registry means.

Scan output

Turns repository evidence into a structured draft batch that reviewers can read quickly.

PR shaping

Routes intake changes into draft pull requests with labels, summaries, and reviewer checklists.

Guardrails

Runs schema, layout, and naming checks so graph changes do not drift across unrelated branches.

Reviewer Path

A reviewer should be able to decide what happens next without reverse-engineering the whole repo.

For intake PRs

  • Confirm the batch belongs under registry-for-review/.
  • Check proposed names against existing canonical skills and named implementations.
  • Ask for clearer evidence when the behavior cannot be reproduced or understood.

For graph PRs

  • Treat registry/gaia.json as the source of truth.
  • Verify prerequisites, rank, evidence class, and status before merge.
  • Keep generated files in sync through the registry scripts, not hand edits.

Branch Shape

Gaia uses narrow feature branches so schema, CLI, docs, design, and review intake can evolve independently. When a proposal changes the meaning of the graph, keep that discussion separate from website copy or tooling polish.
