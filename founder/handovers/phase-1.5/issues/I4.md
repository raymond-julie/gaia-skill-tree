## Definition of Done

- [ ] `.github/workflows/meta-guard.yml` extended with two jobs:
  - **Job A `apex-gate`** — for any PR adding a new `level: 6★` line to a named skill, require `apex-promotion` label + 2 verifier sign-offs (re-uses 4★+ verifier check from `verifier-cluster.py`).
  - **Job B `system-wide-cap`** — query the registry for count of `level: 6★` named skills; fail if count > 5 after this PR's diff applies. Cap value reads from `registry/schema/meta.json` `apexGate.systemWideCap` (set by I1).
- [ ] `scripts/auditApexAtG7.py` runs the 6-active-predicate audit on a single skill ID; prints pass/fail per predicate. Used by reviewers and Job A.
- [ ] `tests/test_meta_guard.py` extended with apex-gate cases (or new test file if none exists).

## Constraints

- Job A must allow `gaia dev reclassify --to 6★` flows that already carry the label + sign-offs (don't break I5 mechanically).
- Job B's count operates on the head-of-PR state. So I5's two demotions (6★→5★) reduce the count from 2 → 0; subsequent promotions are gated by Job A.
- **Read the cap from schema, not hardcoded** so future RFC changes don't require workflow edits.

## Verification

1. Open a draft PR adding a 6★ promotion without `apex-promotion` label → Job A fails.
2. Same PR with label but only 1 verifier sign-off → Job A fails.
3. Same PR with label + 2 sign-offs → Job A passes; Job B then checks system-wide cap.
