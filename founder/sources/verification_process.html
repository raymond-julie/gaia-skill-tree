ev-collection

Data Intake

The collection phase aggregates raw proof vectors from several primary channels. External metadata, package versions, and contributor assertions are harvested and mapped directly to registry slots.

  • Multi-Channel Scraping: Automated collector scripts harvest metrics directly from GitHub API, YouTube showcasing videos, blogs, and arXiv paper citation repositories.
  • Granular Categorization: Raw evidence is classified based on metric type: repo, proxy-containment, arxiv, or social-signal.
  • Master Evidence Consolidation: Merges individual star tiers (Tiers 1★ to 6★) into a central data structure: unified_evidence_lake.md.

ev-star-verification

Star Verification

The validation and live mapping phase validates registry files against live GitHub metrics, resolving repository path counts and generating partitioned raw source tier dumps.

  • Registry Loading & Parsing: Reads evidence data directly from canonical index structures (gaia.json, named-skills.json, and registry/named/).
  • GitHub Live Star Retrieval: Invokes the GitHub CLI (gh repo view) to check and cache live stargazer counts for all primary repositories and proxy-containment targets.
  • Partitioning & Formatting: Separates evidence records into their corresponding star tiers (1★ to 6★) and generates raw markdown files inside founder/sources/data_lake/.
# Invoked via the local ev-star-verification agent skill workflow /ev-star-verification
3

ev-adversarial-audit

Vulnerability Scans

Parallel adversarial subagents stress-test the data lake, analyzing evidence descriptors, repository links, star counts, and proxies from a "Devil's Advocate" perspective to catch quality slippage.

  • Subagent Parallelization: The data lake files are divided among 4 parallel adversarial reviewer subagents to test links and assess qualitative claims.
  • Noise Filtering: Identifies subjective wording ("elite", "best"), procedural rank annotations, or stale database migration notes.
  • Audit Synthesis: A 5th synthesis subagent aggregates findings and appends them to the master source report (e.g. source_report_2026_06_19.md).
# Orchestrated via the local ev-adversarial-audit agent skill workflow /ev-adversarial-audit
4

ev-link-validation

Active API Scrapes

An automated API sweep fetches and scrapes all unique links inside the evidence data lake. Firecrawl's scraping engine resolves HTTP statuses, handles redirects, and verifies actual link availability.

  • Dynamic Link Extraction: The validator script extracts unique links across all tier documents, deduplicating them to prevent redundant sweeps.
  • API Scrape Verification: Hits the Firecrawl scraper API in parallel (limited to 2 concurrency) to check status codes and verify output structural formats.
  • Failures & Outages Logging: Captures timeouts (greater than 45 seconds) and registers HTTP 404 broken links, outputting the results into the official verification log.
